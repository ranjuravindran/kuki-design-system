How this is organized

Three tiers, same as Figma: primitives (raw ramps) → semantic (role-based, light+dark) → components (consume semantic only). Nothing in react/src touches a primitive or a raw hex value directly.

tokens/generate_tokens.py ← single source of truth, asserts WCAG AA on save tokens/css/tokens.css ← generated CSS custom properties (light + dark) react/src/components/*.tsx ← typed components, semantic tokens only react/styles/components.css ← component CSS assets/ai-sparkle-animated.svg ← KuKi's real animated Pet AI icon

Usage guidelines

Do

  • Use semantic tokens (surface/*, text/*) — never raw hex or primitives — in components.
  • Reserve status colors for symptom-free streaks only.
  • Keep buttons as pills; keep cards at radius/container with smoothing on.

Don't

  • Don't introduce a second typeface — Google Sans Flex only, vary by weight.
  • Don't use pure black for high-emphasis fills — use chocolate ink.
  • Don't invent new status hues — extend the existing five-state model.

Typography

Google Sans Flex, one family across weights ExtraLight–Bold. No secondary typeface, no code font — this is a pet health app, not a code editor. This section is theme-aware; toggle dark mode to see both.

Components

Buttons

One unified Button component — the old separate IconButton is now just iconOnly. 5 appearances × 2 tones × 6 states (incl. loading) × 3 sizes × icon-only × selected = 720 variants in Figma. A representative set below; full matrix on the Button page.

Appearance

<Button appearance="primary" size="lg">Start tracking</Button> <Button appearance="secondary">View as Guest</Button> <Button appearance="ghost">Skip</Button> <Button appearance="link">Learn more</Button> <Button appearance="emphasis">Get started</Button>

Icon slots

4 independent slots: outer leadingIcon/trailingIcon sit at the button's own edge padding; inner leadingAdornment/trailingAdornment hug the label inside their own content group, at a gap that's independent of the outer icons.

<Button appearance="emphasis" size="lg" leadingIcon={<HomeIcon/>} leadingAdornment={<CheckIcon/>} trailingAdornment={<CheckIcon/>} trailingIcon={<HomeIcon/>}> Content </Button>

Icon scaling by size

Icons are bound to distinct size tokens per button size (12 / 16 / 20px labeled, 16 / 20 / 24px icon-only) — not a single fixed size stretched to fit.

Tone, disabled & loading

loading is never an overlay on the normal appearance — it forces the same blocked look as disabled, with a spinner as the leading adornment next to editable text.

<Button appearance="primary" tone="destructive">Log seizure</Button> <Button appearance="primary" loading>Please wait</Button> <Button appearance="primary" disabled>Disabled</Button>

Icon-only (iconOnly)

Same appearance/tone/state/loading system as labeled buttons — just iconOnly forcing a fixed circular shape instead of a hug-padded pill. This replaces the old separate IconButton.

<Button appearance="emphasis" size="lg" iconOnly label="Add" icon={<AddIcon/>} /> <Button appearance="ghost" iconOnly selected label="Favourite" icon={<HeartIcon/>} />

Components

Pet AI icon

The swirling background is a CSS approximation of a shader — a blurred, rotating conic gradient screen-blended over a radial base (no WebGL). The sparkle burst on top is KuKi's real animated asset, inlined at runtime with uniquified ids (preview/ai-orb.js) so its CSS keyframes actually run — browsers freeze animated SVGs loaded via plain <img> at their first frame — and so multiple orbs on one page never collide.

Three sparkles fire in sequence on a 1.9s loop, per the source animation's own embedded @keyframes. Respects prefers-reduced-motion for the shader background.
